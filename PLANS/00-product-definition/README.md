# 0단계: 제품 정의 (Product Definition)

## 1) 제품 요구사항 문서(PRD)

### 1.1 제품 개요
- **코드네임**: `win-certbot-gui`
- **제품명(가칭)**: WinCertbot Manager
- **플랫폼**: Windows 10/11 (x64 우선, ARM64는 후속 검토)
- **제품 목적**: Windows 운영자가 CLI 지식 없이도 Let's Encrypt 인증서를 발급/갱신/관리하도록 지원하는 데스크톱 GUI

### 1.2 문제 정의
- 기존 Certbot/ACME 도구는 CLI 기반으로 초보자 진입장벽이 높음
- HTTP-01/DNS-01 선택 및 실패 원인 파악이 어려움
- 와일드카드 인증서 처리 규칙(DNS-01 필수)을 모르면 실패 확률이 높음
- 인증서 만료 관리와 갱신 운영 자동화가 복잡함

### 1.3 목표 및 성공 기준
- **사용성 목표**: 초보 운영자도 마법사 기반으로 10분 이내 첫 인증서 발급
- **기능 목표**: HTTP-01, DNS-01 기반 발급/갱신 플로우 제공
- **안정성 목표**: 주요 실패 원인을 UI에서 분류/안내하고 재시도 경로 제공
- **보안 목표**: 키/토큰 평문 저장 금지 및 민감정보 마스킹

#### KPI (초기)
- 첫 발급 성공률(스테이징 기준) 90% 이상
- 와일드카드 발급 실패 중 사용자 설정 오류 비중 30% 이하
- 갱신 작업 실패 후 사용자 재시도 성공률 80% 이상

### 1.4 대상 사용자
- Windows 기반 소규모/중소 규모 서비스 운영자
- IIS/Nginx/Apache를 운영하지만 ACME 자동화 경험이 적은 사용자
- 멀티 도메인 인증서/와일드카드 인증서가 필요한 DevOps/개발자

### 1.5 사용자 시나리오 (핵심)
1. 사용자가 도메인 목록을 입력한다.
2. 시스템이 wildcard 포함 여부를 감지한다.
3. challenge 방식(HTTP-01/DNS-01)을 유효 범위 내에서 선택한다.
4. 검증 준비(파일 업로드 또는 DNS TXT 등록) 상태를 체크리스트로 확인한다.
5. ACME 검증/발급 완료 후 인증서와 키를 안전 저장하고 결과를 UI에 표시한다.
6. 만료 임박 시 갱신 작업을 실행하고 로그/결과를 확인한다.

### 1.6 기능 요구사항

#### 필수 기능 (MVP)
- 신규 인증서 발급
- 기존 인증서 갱신
- ACME 계정 키 생성/재사용
- 인증서/체인/키 내보내기(PEM)
- HTTP-01, DNS-01 challenge 지원
- wildcard 도메인에서 HTTP-01 선택 차단
- 작업 로그/오류 로그 표시 및 실패 원인 안내

#### 확장 기능 (v1.1 이후)
- DNS 공급자 플러그인 추가 확장
- 인증서 만료 알림 고도화(배지/토스트/시스템 알림)
- 다중 프로파일(서버별 설정 세트) 관리

### 1.7 비기능 요구사항
- **보안**: Windows DPAPI 기반 암호화 저장 우선, 비밀정보 로그 마스킹
- **신뢰성**: 네트워크/DNS 전파 지연 고려한 재시도 정책
- **성능**: 일반 단일 도메인 발급 플로우에서 UI 응답성 유지(작업 중 블로킹 최소화)
- **사용성**: 단계형 마법사 + 상태 기반 안내 문구
- **확장성**: DNS provider 인터페이스 기반 플러그인 구조

### 1.8 정책/규칙
- wildcard(`*.example.com`) 포함 주문은 DNS-01만 허용
- 스테이징 환경을 기본값으로 제공하고 프로덕션 전환 시 명시적 확인
- 민감 키 연산 및 복호화는 Main/Core 레이어에서만 수행

### 1.9 제외 범위 (MVP)
- Linux/macOS 네이티브 지원
- ACME 외 타 인증기관 프로토콜 직접 지원
- 엔터프라이즈 중앙관리(멀티 테넌트, RBAC, 원격 정책 배포)

## 2) 범위 정의 (MVP / v1.1 백로그)

### 2.1 MVP 범위

#### In Scope
- Electron + React + TypeScript 기반 데스크톱 앱 골격
- ACME 기본 플로우: 계정 등록/Order/Challenge/Finalize/인증서 다운로드
- Challenge 방식
  - HTTP-01: 파일 생성/업로드 가이드/사전 HTTP 체크
  - DNS-01: TXT 생성 가이드/자동화(최소 1개 공급자 또는 수동 모드)
- 발급/갱신 결과 대시보드 표시
- 로그 뷰어 및 오류 진단 메시지
- 키/토큰 암호화 저장(DPAPI 우선)
- wildcard DNS-01 강제 규칙

#### Out of Scope
- 다수 DNS 공급자 완전 자동화(초기 1개 + 수동 모드 우선)
- 자동 업데이트 체계 완성
- 대규모 조직용 권한/감사 체계

### 2.2 v1.1 백로그
- DNS 공급자 플러그인 다수 확장(Cloudflare/Route53/AzureDNS 등)
- 갱신 스케줄러 고도화(백그라운드 작업 안정성, 실패 재시도 전략 강화)
- 인증서 배포 연동(예: IIS 바인딩 보조, 후처리 훅)
- 설치/배포 개선(MSI/NSIS 최적화, 업데이트 UX)
- 운영 기능 강화(크래시 리포팅, 진단 번들 내보내기)

### 2.3 릴리즈 게이트 (MVP 완료 기준)
- 단일 도메인 HTTP-01 발급 성공
- 단일 도메인 DNS-01 발급 성공
- wildcard DNS-01 발급 성공
- wildcard에서 HTTP-01 UI 차단 검증 완료
- 키/토큰 평문 저장 금지 검증 완료
- 주요 실패 케이스(HTTP 접근 실패, DNS 전파 지연) 안내 메시지 검증 완료

## 3) 위험요소 및 제약사항 목록

### 3.1 기술적 위험
1. **ACME Rate Limit**
   - 위험: 반복 실패 시 발급 제한 도달
   - 대응: 스테이징 기본값, 재시도 간격 안내, 실패 원인 우선 해결 UX
2. **DNS 전파 지연/불확실성**
   - 위험: TXT 반영 지연으로 검증 실패
   - 대응: 다중 resolver 확인, 대기시간 가이드, 재검증 버튼
3. **HTTP 경로 접근 이슈**
   - 위험: 방화벽/리버스프록시/리다이렉트로 challenge 파일 접근 실패
   - 대응: 사전 URL 체크, 상태/응답코드 노출, 진단 가이드
4. **민감정보 노출**
   - 위험: 로그/설정 파일에 키 또는 API 토큰 노출
   - 대응: 마스킹 정책, 암호화 저장, IPC 최소 권한

### 3.2 제품/운영 위험
1. **초보 사용자 오설정**
   - 위험: 도메인 오타, 잘못된 챌린지 선택
   - 대응: 입력 검증, wildcard 규칙 자동 강제, 맥락형 가이드
2. **외부 의존성 변화**
   - 위험: DNS 공급자 API/ACME 정책 변경
   - 대응: 공급자 어댑터 추상화, 버전 호환 레이어, 문서화된 업데이트 절차
3. **배포 신뢰성**
   - 위험: Windows 보안 정책/백신으로 실행 차단
   - 대응: 코드서명, 설치 패키지 표준화, 배포 전 호환성 테스트

### 3.3 제약사항
- 대상 플랫폼은 Windows 10/11 중심으로 제한
- wildcard는 프로토콜 제약상 DNS-01 필수
- 네트워크/DNS 상태에 따라 발급 시간 변동이 큼
- 초기 버전은 공급자 자동화 범위를 제한할 수 있음
- 민감 연산은 Renderer가 아닌 Main/Core에 국한해야 함
